## 2024-05-24 - Avoid Chained sorting and string allocation in List Operations
**Learning:** In large list sorts and filters (like processing lists of applications), chained `.sortedBy { it.name.lowercase() }.sortedBy { !it.isModified() }` is an anti-pattern. It creates intermediate collections (allocating a new list per call) and `lowercase()` allocates a new string for every item comparison. Similarly, applying `.lowercase()` to both query and item properties in `.filter { ... }` causes repeated allocations.
**Action:** Replace multiple `sortedBy` calls with a single `.sortedWith(compareBy(...).thenBy(...))`. When string property sorts are needed, use `String.CASE_INSENSITIVE_ORDER` (e.g., `thenBy(String.CASE_INSENSITIVE_ORDER) { it.name }`) to avoid allocating `.lowercase()` strings. During filtering/searching, use `.contains(query, ignoreCase = true)` rather than converting items to `.lowercase()`. Be careful when migrating from multiple `.sortedBy` to `.compareBy().thenBy()` that the sort priority sequence is inverted to maintain identical results since `sortedBy` is a stable sort.
